● You're asking the right question, and your hypothesis is mostly correct. Let
  me break down why the field is where it is.

  The screenshot+coordinates approach exists for generality, not quality

  The "computer use" paradigm (screenshot → vision model → click x,y) was
  designed to control any visual interface — native apps, canvas-rendered
  content, Electron apps, remote desktops. Browsers are just one target. If
  you're building one agent that can do everything, pixels are the universal
  API.

  But for web specifically, it's a terrible fit. It's like OCR-ing a PDF that
  you have the source LaTeX for.

  Your hypothesis is largely right — and here's why it hasn't won yet

  The DOM/AST approach is superior for most websites. The blockers are real but
  shrinking:

  1. Context window was the fundamental bottleneck

  A typical modern web page has 10,000–50,000 DOM nodes. Gmail's inbox
  serializes to hundreds of thousands of tokens. Until recently, you couldn't
  fit a meaningful page representation into a model's context. This is the
  single biggest reason screenshot approaches dominated — an image is a
  fixed-cost compression of the entire visual state. With 1M+ context windows,
  this constraint is loosening fast.

  2. The DOM lies about what's visible and interactive

  Raw HTML doesn't tell you:
  - What's actually rendered (display:none, opacity:0, off-screen positioning,
  z-index stacking)
  - What's clickable (event listeners attached by JS, not onclick attributes —
  React, Vue, etc. bindprogrammatically)
  - What's in the viewport vs. scrolled away
  - Visual grouping (flexbox/grid layout doesn't match DOM nesting order)

  So you can't just parse the HTML and enumerate "clickable things." You need
  the computed state — which the browser knows, but the raw source doesn't
  express.

  3. Modern frameworks generate semantic garbage

  Your "90% have clean AST" is true for content sites, blogs, docs, e-commerce.
  But the apps where agents would be most valuable — dashboards, SaaS tools,
  admin panels — are often:
  - Deeply nested <div> soup from React/Vue component trees
  - Shadow DOM that hides internals
  - Virtual scrolling (only renders visible rows — the "data" isn't in the DOM
  at all)
  - Dynamic content loaded via JS after initial page load (the HTML is an empty
  shell + a <script> tag)

  The actual right answer: the accessibility tree

  There's a middle path that validates your intuition. Browsers compute an
  accessibility tree (a11y tree) — it's what screen readers use. It:
  - Contains only interactive/meaningful elements (buttons, links, inputs,
  headings)
  - Has semantic roles and names (button "Submit", link "Settings", textbox 
  "Search")
  - Reflects computed visibility and enabled state
  - Is dramatically smaller than the full DOM (hundreds of nodes instead of tens
   of thousands)
  - Is already designed to answer exactly the question "what can a user do on
  this page?"

  This is essentially the structured, semantic representation you're describing.
   Some newer agent frameworks (Browser Use, Stagehand, AgentQL) do use this or
  similar DOM-pruning approaches, and they outperform screenshot-based agents
  for web tasks.

  The hybrid answer

  The best web agents emerging now do roughly:
  1. A11y tree / pruned DOM for understanding page structure and available
  actions (your approach)
  2. Network interception for understanding API calls (your point about REST
  being readable)
  3. Vision as fallback for visually complex layouts, canvas content, CAPTCHA,
  or when the DOM is genuinely opaque

  The screenshot-first paradigm persists because (a) it was general-purpose and
  shipped first, (b) "one model, one approach" is simpler to build and demo, and
   (c) the context window problem was real until very recently. But you're right
   that for the vast majority of the web, reading the code is strictly better
  than looking at the picture.

● That's a sharp counterpoint, and it basically defeats half of what I said.

  Vimium runs as a content script inside the browser context. It calls
  querySelectorAll for interactive selectors (a, button, input, [role="button"],
   [onclick], etc.), checks getComputedStyle() for visibility, checks
  getBoundingClientRect() for viewport position, and it's done. A few hundred
  lines of JS. Works on Gmail, Jira, GitHub, Slack — the "hard" apps.

  So all my points about "the DOM lies about visibility" and "you can't tell
  what's clickable" — Vimium solves those with trivial browser APIs that have
  existed for a decade. The computed DOM isn't the raw HTML; it's the live,
  resolved state. The browser already did the hard work.

  The honest answer is: there's no fundamental technical reason agents can't do 
  what Vimium does. It's a product and momentum problem:

  1. "Computer use" was built for desktop generality (controlling any app), then
   got shoe-horned onto browsers where it's the wrong tool
  2. The screenshot approach demos well and is conceptually simple to ship
  3. Once the research labs published the screenshot paradigm, everyone
  cargo-culted it

  An agent that does Vimium-style enumeration + injects text labels (inner text,
   aria-label, placeholder) alongside each element gives the LLM a compact,
  semantic, already-solved representation of "what can I do on this page." No
  vision model needed, no coordinate math, no pixel-level errors.

  You're right. The engineering is trivial. The industry is just pointed in the
  wrong direction.

● Good idea. Let me think through the architecture with you.

  The core insight

  Vimium's entire value is one function: enumerate what's actionable, label it, 
  let the user pick. An MCP server that does this gives an agent the same
  superpower — but instead of visual overlays, it returns a text list the LLM
  can reason about.

  Architecture

  Chrome (user's browser, --remote-debugging-port=9222)
      ↕ CDP WebSocket
  MCP Server (Node.js, stdio)
      ↕ stdio
  Claude Code / any MCP client

  No custom extension needed. CDP lets you inject JS into any tab on a running
  Chrome instance. You already have Chrome running with debugging enabled (your
  chrome-devtools-mcp proves that).

  The tool surface — three tools, that's the MVP

  scan — the Vimium f key. Injects JS that:
  - querySelectorAll for interactive selectors (a[href], button, input, select,
  textarea, [role="button"], [onclick], [tabindex], etc.)
  - Filters by getComputedStyle (visibility) and getBoundingClientRect (in
  viewport)
  - Extracts semantic label: aria-label || textContent || placeholder || title
  - Stores element references in window.__agentElements[] for later
  - Returns a compact list:

  [1] button "Sign In"
  [2] input[email] placeholder="Email"
  [3] input[password] placeholder="Password"
  [4] link "Forgot password?" → /reset
  [5] link "Create account" → /signup

  That's ~50 tokens for a login page. A screenshot of the same page costs ~1,500
   tokens and requires vision.

  act — click, type, select, toggle. One tool, action dispatched by type:
  {"id": 1, "action": "click"}
  {"id": 2, "action": "type", "text": "user@example.com"}

  Retrieves element from window.__agentElements[id], acts on it. Re-scans
  automatically after acting and returns the new state.

  read — get text content of the page or a specific region. For when the agent
  needs to read results, not interact. Extracts the main content (using <main>,
  <article>, or largest text block heuristic), returns clean text.

  That's it. Three tools. Scan, act, read.

  Key design decisions

  Why CDP, not an extension?
  - Zero install friction — just connect to Chrome's debug port
  - You already run Chrome with --remote-debugging-port
  - No Chrome Web Store, no manifest, no permissions dialogs
  - CDP gives you tab management for free

  Why store references in window.__agentElements instead of CSS selectors?
  - Direct object references — no re-querying, no fragile selector paths
  - Works even if DOM mutates between scan and click
  - One line to act: window.__agentElements[3].click()
  - Re-scan invalidates the old array, so stale IDs fail cleanly

  Why re-scan after every action?
  - Pages change after interaction (modals open, navigation happens, content
  loads)
  - Agent always has fresh state
  - Matches how a human uses Vimium: press f, pick, press f again

  Why one act tool instead of click + type + select?
  - Fewer tools = less decision overhead for the LLM
  - The element's tag already implies valid actions
  - Keeps the MCP surface minimal

  Tech stack

  - @anthropic-ai/sdk + @modelcontextprotocol/sdk — MCP server in Node
  - chrome-remote-interface — lightweight CDP client (no puppeteer bloat)
  - TypeScript, single file for MVP

  What the agent loop looks like

  Agent: scan()
  → [1] button "Login"  [2] link "Sign up"  [3] input[email] ...

  Agent: act(3, "type", "me@example.com")
  → typed. rescanned:
    [1] button "Login"  [2] link "Sign up"
    [3] input[email] value="me@example.com"  [4] input[password] ...

  Agent: act(4, "type", "hunter2")
  → typed. rescanned: ...

  Agent: act(1, "click")
  → clicked. navigated to /dashboard. rescanned:
    [1] link "Settings"  [2] link "Profile" ...

  Clean, deterministic, no coordinate math, no vision model, sub-100 tokens per
  exchange.

  Revised tool design: affordance-typed interactions

  You're right that the LLM shouldn't decide "do I click or type this." Vimium
  already classifies elements internally — isEditable(), isSelectable(),
  isFocusable() — it just doesn't expose that to the user because a human
  already knows. We expose it explicitly.

  scan() returns elements grouped by what you can do with them:

  Page: GitHub Settings
  URL: https://github.com/settings/profile

  PRESS → use press(id)
    [1] link "Profile"
    [2] button "Update profile"
    [3] link "Notifications"

  TYPE → use type(id, text)
    [4] input "Name" value="wookie"
    [5] textarea "Bio" placeholder="Tell us about yourself"

  SELECT → use select(id, value)
    [6] select "Language" value="English" options=["English","Spanish","French"]

  TOGGLE → use toggle(id)
    [7] checkbox "Available for hire" ✓

  press(id) — clicks/activates. Validates element is pressable. Returns
  auto-rescan.

  type(id, text) — types into field. Validates element is typeable. Clears first
   by default. Returns new value.

  select(id, value) — picks option. Validates element is selectable. Returns new
   value.

  toggle(id) — flips checkbox/radio/switch. Returns new state.

  read() — extracts page text content (non-interactive). For reading results,
  articles, error messages.

  Each tool rejects invalid affordances with a clear error: "Element [2] is a 
  button — use press(), not type()". Structurally impossible to misuse.

  What we borrow from Vimium

  The installed extension is at ~/.config/chromium/Default/Extensions/dbepggeogb
  aibhgnhhndojpepiihcmeb/2.4.2_0/. The key logic:

  - Element discovery from link_hints.js — the clickability detection (native
  tags, role attributes, onclick, jsaction, Angular bindings, tabindex,
  scrollable containers)
  - Visibility filtering from dom_utils.js — getVisibleClientRect()
  (getClientRects, min 3x3px, viewport crop, computed visibility)
  - Element classification — isEditable(), isSelectable() → maps directly to our
   affordance types
  - Text extraction — generateLinkText() (labels, img alt, textContent, title)
  - False positive filtering — removes nested clickable elements (child button
  inside clickable div)

  We strip the hint rendering, keyboard handling, and all UI — keep only the
  detection + classification engine.
